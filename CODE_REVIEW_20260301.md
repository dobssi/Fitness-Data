### 4. Workflow duplication across 4 athlete pipelines (1,171 lines)
**Severity: Medium — maintenance burden**

`ian_pipeline.yml` (332 lines) and `steve_pipeline.yml` (332 lines) differ by only ~40 lines after normalising athlete names. `nadi_pipeline.yml` (223 lines) is shorter but follows the same structure. `pipeline.yml` (284 lines) for Paul has a different structure (Stryd mode) but shares 60%+ of the steps.

Every pipeline change (e.g. adding a new step, fixing a dependency) needs to be applied 3-4 times. This is already causing drift — Nadi's workflow is 109 lines shorter than Ian/Steve's.

**Options:**
- **Short term:** Accept duplication, use a checklist when changing workflows.
- **Medium term:** Extract shared steps into a reusable composite action (`.github/actions/run-pipeline/action.yml`) parameterised by athlete name, config path, and mode. Each athlete workflow becomes ~30 lines calling the shared action.
- **Long term:** Single workflow with a matrix strategy iterating over athletes.
